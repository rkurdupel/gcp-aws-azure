## Architecture

```
User
  ↓
app.coin-ops.pp.ua (Cloudflare DNS)
  ↓
AWS Application Load Balancer (port 80)
  ↓ health check: GET /health every 30s
  ├── app-1 VM (10.10.1.12)         app-2 VM (10.10.1.13)
  │   nginx + ui + proxy + history   nginx + ui + proxy + history
  └──────────────────┬───────────────────────────┘
                     ↓
               db VM (10.10.1.11)
               postgres + rabbitmq + redis

SSH access:
  your laptop → bastion (public IP) → app-1 / app-2 / db
```

### AWS Resources Created by Terraform

| Resource | Purpose |
|---|---|
| VPC `10.10.0.0/16` | Isolated network for all resources |
| Public subnet `10.10.0.0/24` | Bastion lives here, has direct internet |
| Public subnet `10.10.2.0/24` | Second AZ for ALB (ALB needs 2 AZs minimum) |
| Private subnet `10.10.1.0/24` | App and db VMs, no direct internet access |
| Internet Gateway | Single exit point for the whole VPC |
| NAT Gateway | Lets private VMs reach internet (apt update, docker pull) |
| Elastic IP | Fixed public IP for NAT Gateway |
| Bastion EC2 | SSH jump host — only entry point for admin access |
| db EC2 | Runs postgres, rabbitmq, redis |
| app-1, app-2 EC2 | Runs nginx, ui, proxy, history |
| Application Load Balancer | Distributes traffic across app VMs |
| Target Group | Pool of app VMs with health checks |
| Security Groups | Firewall rules per VM role |
| Cloudflare DNS CNAME | Maps domain to ALB |

### Security Groups

```
bastion-sg:  allow SSH port 22 from your IP only
app-sg:      allow HTTP port 80 from ALB only, SSH from bastion only
db-sg:       allow postgres/rabbitmq/redis from app VMs, SSH from bastion
lb-sg:       allow HTTP port 80 from internet (0.0.0.0/0)
```

## Project Structure

```
coinops/
  config/
    config.yml                     ← master config: VMs, network, firewall

  terraform/
    main.tf                        ← entry point, calls aws/gcp modules
    locals.tf                      ← reads config.yml
    outputs.tf                     ← bastion IP, ALB DNS, SSH config, inventory
    backend.tf                     ← S3 state bucket + DynamoDB locking
    provider.tf                    ← AWS, GCP, Cloudflare providers
    versions.tf                    ← provider version constraints
    modules/
      aws/
        main.tf                    ← creates all AWS resources
        outputs.tf
        variables.tf
        firewall/                  ← security groups
        network/                   ← VPC, subnets, IGW, NAT, route tables
        vm/                        ← EC2 instances
        load_balancer/             ← ALB, target group, listener
      gcp/                         ← GCP equivalent (switch cloud: gcp in config)

  ansible/
    ansible.cfg                    ← SSH key, host key settings
    inventory.cloud                ← AUTO-GENERATED by post-apply.sh, never edit
    cloud-provision.yml            ← installs Docker on all VMs
    cloud-deploy.yml               ← deploys app on VMs
    group_vars/
      all/main.yml                 ← shared variables read from .env
    templates/
      cloud-nginx.conf.j2          ← nginx routing config
      cloud-app.compose.yaml.j2   ← docker compose for app VMs
      cloud-db.compose.yaml.j2    ← docker compose for db VM
      cloud-app.env.j2             ← .env for app containers
      cloud-db.env.j2              ← .env for db containers

  scripts/
    bootstrap-aws.sh               ← one-time AWS setup (IAM, S3, DynamoDB)
    post-apply.sh                  ← updates SSH config + inventory after apply
    lab.sh                         ← one command deploy pipeline
    verify_failover.sh             ← stops 1 VM, confirms failover works

  ui/                              ← Flask UI service
  proxy_service/                   ← Flask proxy (fetches currency prices)
  history_service/                 ← Go history service (saves to postgres)
  docker-compose.yml               ← local development only
```

## What Each File Does

### `config/config.yml`
Master configuration file. Everything reads from here. Terraform uses it to create infrastructure. Change `cloud: aws` to `cloud: gcp` to deploy to GCP instead.

Key sections:
- `instances` — list of VMs with roles and private IPs
- `network` — VPC and subnet CIDRs
- `firewall.ssh_source_ranges` — your IP for SSH access (run `curl ifconfig.me`)
- `catalog` — VM sizes and OS images per cloud

### `terraform/locals.tf`
Reads `config.yml` with `yamldecode()` and makes values available as `local.config.*` throughout Terraform. Central place that connects config to infrastructure.

### `terraform/outputs.tf`
After `terraform apply`, exposes:
- `bastion_public_ip` — to SSH into bastion
- `alb_dns_name` — to test app without domain
- `ssh_config` — SSH shortcuts written to `~/.ssh/coinops-aws.generated`
- `ansible_inventory` — host list with IPs written to `ansible/inventory.cloud`

### `scripts/post-apply.sh`
Must run after every `terraform apply`. Does:
1. `terraform output ansible_inventory` → saves to `ansible/inventory.cloud`
2. `terraform output ssh_config` → saves to `~/.ssh/coinops-aws.generated`
3. Adds `Include ~/.ssh/coinops-aws.generated` to `~/.ssh/config`

After this runs you SSH by name: `ssh coinops-bastion`, `ssh coinops-db`

### `ansible/cloud-provision.yml`
Installs Docker on all VMs (bastion, db, app-1, app-2). Runs two roles:
- `common` — apt update, install packages, create user, set timezone to UTC
- `docker` — add Docker GPG key, add Docker repo, install Docker, add user to docker group

### `ansible/cloud-deploy.yml`
Deploys the app in order:
1. Creates `/opt/coinops/cloud-db` and `/opt/coinops/cloud-app` directories
2. Renders templates and copies to VMs
3. Copies source code (ui, proxy_service, history_service) to app VMs
4. `docker compose up` on db VM → starts postgres, rabbitmq, redis
5. `docker compose up --build` on app VMs → builds and starts nginx, ui, proxy, history
6. Restarts nginx to pick up latest config
7. Waits for `GET /health` to return 200

### `templates/cloud-nginx.conf.j2`
nginx routing on each app VM:
- `GET /health` → returns `{"status":"ok"}` directly (ALB health check uses this)
- `GET /api/chart-data` → forwards to `ui:5000` (exact match, takes priority)
- `GET /api/*` → forwards to `proxy:5001`
- `GET /history-api/*` → forwards to `history:5002`
- `GET /*` → forwards to `ui:5000`

### `scripts/verify_failover.sh`
Tests that the app survives a VM failure:
1. Confirms app is healthy (baseline)
2. Stops one app VM via AWS CLI
3. Waits 60s for ALB to detect it as unhealthy
4. Confirms app still responds (other VM handles traffic)
5. Restarts the stopped VM

### `scripts/lab.sh`
One command pipeline:
```bash
./scripts/lab.sh full     # terraform + post-apply + ansible provision + deploy
./scripts/lab.sh apply    # terraform only
./scripts/lab.sh deploy   # ansible only
./scripts/lab.sh verify   # run failover test
```

## Step by Step Deploy

### Quick deploy (one command)

```bash
source .env
./scripts/lab.sh full
```

### Manual step by step

**Step 1 — Bootstrap AWS (one time only)**
```bash
bash scripts/bootstrap-aws.sh
source .aws-bootstrap.env
```
Creates: IAM user with correct permissions, S3 bucket for Terraform state, DynamoDB table for state locking (prevents two people running terraform apply simultaneously).

**Step 2 — Terraform: create all infrastructure**
```bash
cd terraform
terraform init        # download providers
terraform plan        # preview what will be created
terraform apply       # create resources (~10 min)
cd ..
```

**Step 3 — Generate SSH config and Ansible inventory**
```bash
./scripts/post-apply.sh
```
After this: `ssh coinops-bastion` works without specifying IP or key.

**Step 4 — Install Docker on all VMs**
```bash
source .env
ansible-playbook -i ansible/inventory.cloud ansible/cloud-provision.yml
```
Takes ~5 min. Installs Docker on all 4 VMs.

**Step 5 — Deploy the app**
```bash
ansible-playbook -i ansible/inventory.cloud ansible/cloud-deploy.yml
```
Takes ~10 min. Builds Docker images on VMs and starts all containers.

**Step 6 — Verify deployment**
```bash
# Health check
curl http://app.coin-ops.pp.ua/health
# {"status": "ok"}

# Prices
curl http://app.coin-ops.pp.ua/api/price/BTC
# {"price": 79786.55}

# Open in browser
open http://app.coin-ops.pp.ua
```

**Step 7 — Verify failover**
```bash
./scripts/verify_failover.sh
```

**Step 8 — Destroy when done (saves money)**
```bash
cd terraform && terraform destroy
```

```bash
# Rebuild and restart on all app VMs
ansible-playbook -i ansible/inventory.cloud ansible/cloud-deploy.yml
```

### Check ALB target health
```bash
aws elbv2 describe-target-health \
  --target-group-arn YOUR_TARGET_GROUP_ARN \
  --query 'TargetHealthDescriptions[].[Target.Id,TargetHealth.State]' \
  --output text \
  --region eu-central-1
```

## FAQ

**Q: Why do we need a bastion VM?**
App and db VMs are in a private subnet with no public IP — unreachable from internet directly. Bastion is the only VM with a public IP. You SSH into bastion first, then from bastion into private VMs. Even if someone gets your SSH key, they can only connect from your IP (set in `firewall.ssh_source_ranges` in config.yml).

**Q: Why do we need a NAT Gateway?**
Private VMs have no public IP so they cannot initiate outbound connections to the internet. NAT Gateway lets them make outbound requests (apt update, docker pull images) without being reachable from the internet. Flow: private VM → NAT Gateway → Internet Gateway → internet. Without NAT, Docker cannot pull images and apt cannot install packages on private VMs.

**Q: Why two app VMs instead of one?**
If one VM crashes, the other handles all traffic. ALB detects unhealthy VMs every 30 seconds via health checks and stops sending traffic to them. With only one VM, any crash = complete outage for all users.

**Q: Why does the ALB need subnets in 2 availability zones?**
Each AZ is a physically separate data center. AWS requires ALB to span at least 2 AZs so if one data center has issues the ALB stays operational via the other AZ. Without 2 AZs, `terraform apply` fails with "At least two subnets in two different Availability Zones must be specified".

**Q: What is an ARN?**
Amazon Resource Name — unique identifier for every AWS resource across all services and regions. Format: `arn:aws:service:region:account:resource`. Used for cross-service references, e.g. attaching EC2 instances to a target group requires the target group ARN not just its name.

**Q: Why does `count = local.cloud == "aws" ? 1 : 0` use `[0]` to access the module?**
`count` makes Terraform treat the module as a list even when only 1 is created. `count = 1` creates a list with one item, accessed with `[0]` (first item). `count = 0` creates an empty list (module not created). This lets you conditionally create modules based on which cloud you're deploying to.

**Q: What does `post-apply.sh` do and when to run it?**
Every `terraform apply` may recreate VMs with new IPs (especially bastion after destroy+apply). `post-apply.sh` reads the new IPs from Terraform outputs and updates your local SSH config and Ansible inventory file. Always run it after `terraform apply`. `lab.sh` runs it automatically.

**Q: Why does nginx have a `/health` endpoint returning hardcoded JSON?**
The ALB health check calls `GET /health` on every VM every 30 seconds. nginx returns `{"status":"ok"}` with 200 directly without forwarding to any backend — fast and reliable. `access_log off` prevents thousands of health check entries flooding your nginx logs.

**Q: Why are there two separate .env files (cloud-app.env and cloud-db.env)?**
Each VM only gets the secrets it needs (principle of least privilege). The db VM needs initialization variables (postgres user/password to create the database on first start). The app VMs need connection strings (host/port/credentials to connect). Mixing them would give db VM access to app routing secrets and vice versa.

**Q: What is `hostvars[groups['db'][0]]['ansible_host']` in templates?**
`groups['db']` returns a list of hostnames in the db group: `['coinops-db']`. `[0]` gets the first: `'coinops-db'`. `hostvars['coinops-db']` returns a dictionary of all facts about that host. `['ansible_host']` gets its IP: `10.10.1.11`. This is how app VMs get the db VM's IP in their `.env` files dynamically without hardcoding.

**Q: Why do I need to click a coin twice sometimes?**
With 2 app VMs, ALB alternates requests between them by default. Click 1 → app-1, Click 2 → app-2. Each VM has its own Flask session. Sticky sessions (`stickiness` in target group config) fix this — once assigned to a VM, all your requests in that session go there.

**Q: What does `restart: unless-stopped` mean in Docker Compose?**
Restart the container if it crashes or the VM reboots — but NOT if you manually ran `docker compose down`. Other options: `always` (restarts even after manual stop), `on-failure` (only on crash exit code), `no` (never restart automatically).

**Q: Why use Ansible roles instead of putting all tasks in one playbook?**
Roles separate WHERE to run (defined in playbook with `hosts:`) from WHAT to run (defined in role tasks). The same `docker` role installs Docker on any group of hosts — bastion, db, app VMs. Without roles you'd copy-paste the same tasks for each host group.

**Q: How does `ssh coinops-db` work without specifying IP or key?**
`post-apply.sh` generates `~/.ssh/coinops-aws.generated` with Host entries for all VMs including `ProxyJump coinops-bastion` for private VMs. It also prepends `Include ~/.ssh/coinops-aws.generated` to `~/.ssh/config`. SSH reads this file automatically on every connection.

**Q: Why use Cloudflare for DNS instead of Route 53?**
Both work. Cloudflare is free and gives you DDoS protection and CDN on top of DNS. Route 53 costs $0.50/hosted zone/month. For this project Cloudflare is simpler since the domain was already on nic.ua — just point nameservers to Cloudflare.

**Q: How much does this cost on AWS per hour?**
- 4x t3.micro EC2: ~$0.021/h each = $0.084/h
- NAT Gateway: ~$0.045/h
- ALB: ~$0.008/h
- Elastic IP: ~$0.005/h
- Total: ~$0.14/h (~$3.36/day)

Always run `terraform destroy` when not using it.

**Q: Why does history service fail with "SSL is not enabled on the server"?**
The Go postgres driver requests SSL by default. Our postgres container runs without SSL configured (we add `command: postgres -c ssl=off` in docker compose). Adding `sslmode=disable` to the connection string in `main.go` tells the driver not to use SSL.

**Q: Why does Ansible show `changed=0` when I expect changes?**
Ansible modules are idempotent — they only make changes if needed. `changed=0` means the desired state already exists. If files weren't actually copied, check YAML indentation (must use spaces not tabs) and run with `-v` flag for verbose output.
